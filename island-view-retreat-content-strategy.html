Technical Implementation

How to build 100+ pages efficiently in your existing Next.js 16 codebase.

Your Architecture is Already Set Up for This

You're running Next.js 16 with App Router, SSR, Prisma/Supabase, and you already have a blog system with per-page TSX files, a dynamic campsite system, and good SEO metadata patterns. The codebase is ready — you just need a content scaling strategy.

Option A: Static TSX Pages (Current Approach)

Keep writing each page as its own page.tsx. Works for 20-30 pages but doesn't scale to 120+.

Pros: Full control, unique layouts, no CMS needed

Cons: Slow to produce, hard to maintain consistency, developer bottleneck

Option B: MDX + Content Layer (Recommended)

Add MDX support. Write content in markdown, render with shared templates. Metadata in frontmatter, content in markdown, components inline.

Pros: Fast content creation, consistent templates, non-dev can write

Cons: Initial setup, less layout flexibility per page

Structure: /content/blog/*.mdx → /blog/[slug]/page.tsx

Option C: Database-Driven (Programmatic SEO)

Store content in Supabase. Dynamic [slug] routes render from DB. You already do this for campsites.

Pros: Maximum scale, CMS-like editing, programmatic generation

Cons: More setup, need admin UI, content feels templated if not careful

Extend Prisma: model BlogPost { slug, title, content, keywords, cluster, published }

Option D: Hybrid (Best of Both)

Pillar pages as TSX (full custom design), supporting articles as MDX or DB-driven. Use-case and proximity pages as templates with variable data.

Pros: Quality where it matters, speed where it counts

Cons: Two systems to maintain

TSX for: home, cottage, big-gull-lake, pricing, reviews
Template for: use-case, proximity, seasonal, comparison articles
